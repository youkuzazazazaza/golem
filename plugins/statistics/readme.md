# Statistics 插件

消息统计 + 群成员人物画像插件：自动记录群聊消息、提供发言排行和详情查询，并支持基于历史发言生成「人物画像」。

## 功能特性

- **自动记录**：实时记录所有群聊消息，包括消息类型、发送者、时间等信息
- **发言排行**：支持查看今日、昨日、本周、本月和总排行榜
- **详细统计**：按消息类型统计（文本、图片、语音、视频等）
- **个人详情**：查询指定成员的详细发言数据
- **人物画像**：发「人物画像 <昵称>」基于历史发言生成/更新成员画像（复用 `ai` 插件的 `ai.chat` 能力）
- **三个数据库**：
  - `analysis.db`：存储聚合统计数据，快速查询排行
  - `statistics.db`：存储完整消息记录
  - `profile.db`：存储人物画像（`profiles` 表），与消息库分离互不影响
- **零配置运行**：开箱即用，无需任何配置

## 快速开始

### 1. 启用插件

插件启用后会自动开始记录群聊消息，无需额外配置。

### 2. 查看排行

在群聊中发送以下关键词即可查看对应的排行榜：

- `今日排行`：查看今天的发言排行
- `昨日排行`：查看昨天的发言排行
- `本周排行`：查看本周的发言排行（周一到当前）
- `本月排行`：查看本月的发言排行（1号到当前）
- `总排行`：查看历史总排行

### 3. 查看个人详情

在群聊中发送 `发言详情` 即可查看自己的详细发言统计。

## 人物画像

基于群成员的历史发言，调用大模型（经 `ai` 插件的 `ai.chat` 能力）生成/增量更新人物画像，并**渲染为图片**发送（经 `gg` 插件的 `text.to.image` 能力）。
**依赖**：`ai` 插件（提供 LLM）；`gg` 插件（提供 `text.to.image`，把画像文本渲染成 PNG）；数据源为本插件记录的 `statistics.db`。若 `gg` 未启用，自动回退为纯文本发送。

### 触发形式

| 消息 | 范围 | 说明 |
|------|------|------|
| `人物画像` | 群聊 | **查自己**（群内未指定成员时默认发起人自己） |
| `人物画像` | 私聊 | 仅主人可用，查自己 |
| `人物画像 张三` | 当前群 | 群内按昵称/群显示名/备注/wxid 匹配 |
| `人物画像 @张三` | 当前群 | @ 提人，用 Reminds 里的 wxid 直接定位（最可靠） |
| `人物画像@张三` | 当前群 | 同上（@ 直接跟，无空格也触发） |
| `人物画像 张三 --global` | 跨群 | 全局画像（按联系人缓存匹配，多为好友） |
| `人物画像 张三 --rebuild` | 当前群 | 忽略已有画像，从头冷启动 |

> 前缀「人物画像」后必须紧跟**空白**或**@**才触发（兼容微信 @ 提人插入的非常规空格如 NBSP）；
> 形如「人物画像张三」「人物画像功能真不错」不触发，放行给 ai。

### 权限

| 场景 | 可查询范围 |
|------|-----------|
| 群聊 | 任意群成员均可触发，仅限当前群成员 |
| 私聊 | **仅主人**可查（需配置 `Owner`） |
| 主人画像 | 仅主人本人可查 |

### 生成流程

```
记录该消息 → 解析触发语 + @ 的 wxid → 权限/范围判定 → 读历史发言
   → 按 token 切块 → 每块调 ai.chat 产出「局部观察」 → 合并(已有画像+观察+量化指标) → 落库 profiles 表
```

- **增量水位线**：已生成过的画像只分析 `last_msg_id` 之后的新发言，省 token
- **无新发言**：直接返回已有画像，不空跑 AI
- **消息过多**：查询层只取最近 `2000` 条；切块层超 `30` 块时丢弃最早的、只保留最新 30 块
- **画像维度**：性格与语言风格、兴趣话题、活跃时段、口头禅/表情习惯、与其他成员的关系

### ⚠️ Token 用量提醒

冷启动最费 token：最多 30 块 × 每块 ~4000 token + 一次合并 ≈ **12 万+ input tokens / 人**。
建议冷启动用便宜模型（DeepSeek/Qwen 等，靠 `ai` 插件配置），冷启动后日常只做增量更新（通常 1 次调用），
不要频繁 `--rebuild`。多人/多群会线性放大成本。

### 配置（可选）

插件零配置即可运行；如需调节画像参数，编辑 `plugins/config.toml` 的 `[statistics.config]` 段（首次运行由宿主自动生成默认值）：

```toml
[statistics.config]
chunk_token_budget = 4000       # 每块 token 估算上限；中文模型可调大到 6000
max_single_msg_chars = 2000     # 单条消息硬截断长度
cold_start_max_messages = 2000  # 冷启动最多处理的消息条数
cold_start_max_chunks = 30      # 冷启动最多分块数（超出丢弃最早的）
render_image = true             # 画像是否渲染成图片发送（gg 插件 text.to.image）；false 发纯文本
```

留空（或值为 0）则回退到上列默认值。`render_image` 默认 `true`（渲染图片）；不想看图片时设为 `false` 即发纯文本，gg 未启用或渲染失败时也会自动回退文本。

## 数据存储

### 存储位置

数据库文件默认存储在插件所在目录：
- `analysis.db`：聚合统计数据库
- `statistics.db`：完整消息记录数据库
- `profile.db`：人物画像数据库（`profiles` 表，独立于消息库）

### 数据库结构

#### analysis.db - 聚合统计表
```sql
CREATE TABLE analysis (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    sender TEXT NOT NULL,        -- 群聊 ID
    member TEXT NOT NULL,        -- 成员 ID
    type TEXT NOT NULL,          -- 消息类型
    count INTEGER DEFAULT 1,     -- 当日消息数
    date TEXT NOT NULL,          -- 统计日期
    UNIQUE(sender, member, type, date)
);
```

#### statistics.db - 完整记录表
```sql
CREATE TABLE statistics (
    id INTEGER PRIMARY KEY,      -- 消息 ID
    type TEXT NOT NULL,          -- 消息类型
    content TEXT,                -- 消息内容
    sender TEXT,                 -- 群聊 ID
    receiver TEXT,               -- 接收者 ID
    member TEXT,                 -- 发送成员 ID
    raw TEXT NOT NULL,           -- 原始消息
    timestamp DATETIME           -- 发送时间
);
```

#### profile.db - 画像表（独立库）
```sql
CREATE TABLE profiles (
    chatroom  TEXT NOT NULL DEFAULT '',   -- 群 wxid；全局画像为空
    member    TEXT NOT NULL,              -- 成员 wxid
    profile   TEXT,                       -- 画像文本
    last_msg_id INTEGER DEFAULT 0,        -- 增量水位线（已分析到的最后消息 ID）
    updated_at DATETIME,                   -- 最近更新时间（本地时间）
    PRIMARY KEY (chatroom, member)
);
```

### 数据特点

- **自动去重**：相同消息 ID 只记录一次
- **按日聚合**：analysis 表按天聚合统计，查询速度快
- **完整保留**：statistics 表保留所有消息详情，支持深度分析
- **索引优化**：sender 字段建立索引，加速查询

## 工作原理

### 消息记录流程

```
1. 接收群聊消息
   ↓
2. 提取消息信息（ID、类型、发送者等）
   ↓
3. 写入 statistics 表（完整记录）
   ↓
4. 更新 analysis 表（增加当日计数）
   ↓
5. 继续传递给下一个插件
```

### 排行查询流程

```
1. 接收排行关键词
   ↓
2. 根据时间范围构建 SQL 过滤条件
   ↓
3. 从 analysis 表查询聚合数据
   ↓
4. 查询每个成员的消息类型分布
   ↓
5. 构建排行榜合并转发消息
   ↓
6. 发送给群聊
```

### 详情查询流程

```
1. 接收"发言详情"关键词
   ↓
2. 识别发送者身份
   ↓
3. 分别查询各时间段的统计数据
   - 今日详情
   - 昨日详情
   - 本周详情
   - 本月详情
   - 总结详情
   ↓
4. 格式化并发送（@发送者）
```

## 时间范围说明

| 关键词 | 时间范围 | SQL 过滤条件 |
|-------|---------|------------|
| 今日排行 | 今天 00:00 至当前时刻 | `date = date('now', 'localtime')` |
| 昨日排行 | 昨天 00:00 至 23:59 | `date = date('now', '-1 day', 'localtime')` |
| 本周排行 | 本周一 00:00 至当前时刻 | 从周一开始计算 |
| 本月排行 | 本月 1 号 00:00 至当前时刻 | `date >= date('now', 'localtime', 'start of month')` |
| 总排行 | 所有历史数据 | 无过滤条件 |

**注意**：
- 时间使用服务器本地时区
- 本周从周一开始（ISO 8601 标准）
- 跨天统计在 00:00 时刻分界

## 插件行为

### 优先级和执行顺序

```toml
Priority = -1 << 31  # 最高优先级（最小值，最先执行）
Next = false         # 处理后终止事件链
AlwaysRun = true     # 始终运行，即使其他插件已处理
```

**特点**：
- **最先执行**：优先级最高，在所有插件之前运行
- **透明记录**：普通消息记录后返回 `handled=false`，事件链继续传给后续插件（如 ai），不影响其他插件
- **消费触发**：排行关键词与「人物画像」触发返回 `handled=true`，终止事件链（避免 ai 重复回复）
- **全量统计**：`AlwaysRun=true` 保证会话被劫持时仍记录所有群聊消息

### 事件订阅

插件订阅 `message` 事件，自动接收所有消息。

### 消息处理逻辑

```go
if 是排行关键词:
    查询并发送排行榜
    返回 true（已处理，但仍传递给下一个插件）
else:
    记录消息到数据库
    返回 recorded（记录成功则为 true）
```

## 性能考虑

### 数据库优化

- **批量插入**：使用 `INSERT OR IGNORE` 和 `ON CONFLICT` 减少冲突
- **索引加速**：sender 字段建立索引
- **分库存储**：聚合和明细分开，互不影响

### 查询优化

- **聚合优先**：排行查询使用 analysis 表，避免全表扫描
- **限制返回**：默认只返回 Top 10，减少数据传输
- **缓存连接**：数据库连接复用，减少开销

### 存储空间

假设一个 500 人的活跃群，每天产生 1000 条消息：

- **analysis 表**：约 500 行/天（每人每天每类型一行），约 50 KB/天
- **statistics 表**：1000 行/天，约 500 KB/天

一年存储约 **200 MB**，存储压力较小。

## 常见问题

### 1. 排行榜为什么是空的？

**可能原因**：
- 插件刚启用，还没有记录数据
- 查询的时间范围内没有消息
- 数据库文件损坏

**解决方法**：
- 等待一段时间后再查询
- 尝试查询"总排行"
- 检查数据库文件是否存在

### 2. 为什么我发的消息没有统计？

**检查清单**：
- 确认在群聊中发送（不支持私聊统计）
- 确认插件已启用
- 查看日志是否有错误

### 3. 如何清空统计数据？

直接删除数据库文件即可：
```bash
rm analysis.db statistics.db profile.db
```

插件会在下次启动时自动重建数据库。

### 4. 排行榜最多显示多少人？

默认显示 Top 10，这是代码中的常量 `defaultRankLimit`。

### 5. 私聊消息会被统计吗？

不会。插件只统计群聊消息（`contact.ContactType_CONTACT_TYPE_CHATROOM`），私聊消息会被忽略。

### 6. 统计数据会占用很多空间吗？

一般不会。一个中等活跃度的群，一年的数据大约 200 MB 左右。如果担心空间问题，可以定期备份并清空旧数据。

### 7. 排行关键词可以自定义吗？

目前不支持。关键词在代码中硬编码：
- 今日排行
- 昨日排行
- 本周排行
- 本月排行
- 总排行
- 发言详情

### 8. 能否导出统计数据？

数据库是标准的 SQLite 格式，可以使用任何 SQLite 工具导出：

```bash
# 导出为 CSV
sqlite3 statistics.db "SELECT * FROM statistics" -csv > messages.csv

# 导出为 JSON
sqlite3 statistics.db "SELECT * FROM analysis" -json > analysis.json
```

## 注意事项

### 1. 隐私保护

- 插件会记录所有群聊消息内容
- 数据库文件包含敏感信息，注意保护
- 在公开环境使用时需告知成员

### 2. 存储管理

- 定期检查数据库文件大小
- 超大群（1000+ 人）需要注意存储空间
- 可以定期备份并清空旧数据

### 3. 性能影响

- 插件会拦截所有消息进行记录
- 对于超大群，可能有轻微性能影响
- 数据库操作是异步的，不会阻塞消息处理

### 4. 数据准确性

- 插件启用前的消息不会被统计
- 插件禁用期间的消息不会被记录
- 确保插件持续运行以保证数据完整性

### 5. 时区问题

- 统计使用服务器本地时区
- 如果服务器时区不正确，统计时间可能有偏差
- 建议确保服务器时区设置正确

## 技术细节

### 并发安全

插件使用 `sync.Mutex` 保护数据库操作，确保并发安全：

```go
p.mu.Lock()
defer p.mu.Unlock()
```

### 错误处理

- 数据库连接失败：插件启动失败
- 记录消息失败：记录警告日志，不影响消息传递
- 查询失败：返回错误信息给用户

### 成员信息获取

插件会尝试获取成员的完整信息（昵称、群昵称）：
1. 优先使用消息中的成员信息
2. 如果可用，调用 chatroom ability 查询
3. 显示优先级：群昵称 [昵称] > 群昵称 > 昵称 > username

### XML 转义

合并转发消息的 XML 内容会自动转义特殊字符，防止 XML 注入。
