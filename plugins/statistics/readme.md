# Statistics 插件

消息统计插件：自动记录群聊消息、提供发言排行和详情查询；并暴露 `statistics.query_messages` 能力供其它插件查询历史发言。

## 功能特性

- **自动记录**：实时记录所有群聊消息，包括消息类型、发送者、时间等信息
- **发言排行**：支持查看今日、昨日、本周、本月和总排行榜
- **详细统计**：按消息类型统计（文本、图片、语音、视频等）
- **个人详情**：查询指定成员的详细发言数据
- **能力暴露**：提供 `statistics.query_messages` 跨插件能力，供 `profile` 等插件查询成员历史发言
- **双数据库**：
  - `analysis.db`：存储聚合统计数据，快速查询排行
  - `statistics.db`：存储完整消息记录，支持详细分析
- **零配置运行**：开箱即用，无需任何配置

## 快速开始

### 1. 启用插件

插件启用后会自动开始记录群聊消息，无需额外配置。

### 2. 查看排行

在群聊中发送以下关键词即可查看对应的排行榜：

- `今日排行`：查看今天的发言排行
- `昨日排行`：查看昨天的发言排行
- `本周排行`：查看本周的发言排行（周一到当前）
- `本月排行`：查看本月的发言排行（1号到当前）
- `总排行`：查看历史总排行

### 3. 查看个人详情

在群聊中发送 `发言详情` 即可查看自己的详细发言统计。

## 能力：statistics.query_messages

本插件暴露 `statistics.query_messages` 能力，供其它插件（如 `profile`、`wordcloud`）查询历史发言。

**入参** `map[string]string`：

| key | 必需 | 含义 |
|---|---|---|
| `member` | 与 chatroom 至少一个 | 成员 wxid；缺省=不限成员（整群/全局） |
| `chatroom` | 与 member 至少一个 | 群 wxid；缺省=跨群全局 |
| `since` | 否 | 本地时间下限（含边界），格式 `2006-01-02 15:04:05` |
| `since_id` | 否 | 只取 id > since_id（增量水位线）；缺省=0 |
| `limit` | 否 | >0 时取最近 limit 条并反转为时间正序；0=不限 |

组合语义：`chatroom`+`member`=群内成员；仅 `chatroom`=整群；仅 `member`=跨群全局。

**返回**：`mime="json"`，JSON 数组（时间正序）：
```json
[{"id":123,"content":"...","timestamp":"2026-07-08 10:51:52"}, ...]
```

> timestamp 为本机本地时间（statistics 表存的就是本地时间），`since` 也按本地时间比较。仅返回 `type='文本'` 的消息。

## 数据存储

### 存储位置

数据库文件默认存储在插件所在目录：
- `analysis.db`：聚合统计数据库
- `statistics.db`：完整消息记录数据库

### 数据库结构

#### analysis.db - 聚合统计表
```sql
CREATE TABLE analysis (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    sender TEXT NOT NULL,        -- 群聊 ID
    member TEXT NOT NULL,        -- 成员 ID
    type TEXT NOT NULL,          -- 消息类型
    count INTEGER DEFAULT 1,     -- 当日消息数
    date TEXT NOT NULL,          -- 统计日期
    UNIQUE(sender, member, type, date)
);
```

#### statistics.db - 完整记录表
```sql
CREATE TABLE statistics (
    id INTEGER PRIMARY KEY,      -- 消息 ID
    type TEXT NOT NULL,          -- 消息类型
    content TEXT,                -- 消息内容
    sender TEXT,                 -- 群聊 ID
    receiver TEXT,               -- 接收者 ID
    member TEXT,                 -- 发送成员 ID
    raw TEXT NOT NULL,           -- 原始消息
    timestamp DATETIME           -- 发送时间
);
```

### 数据特点

- **自动去重**：相同消息 ID 只记录一次
- **按日聚合**：analysis 表按天聚合统计，查询速度快
- **完整保留**：statistics 表保留所有消息详情，支持深度分析
- **索引优化**：sender 字段建立索引，加速查询

## 工作原理

### 消息记录流程

```
1. 接收群聊消息
   ↓
2. 提取消息信息（ID、类型、发送者等）
   ↓
3. 写入 statistics 表（完整记录）
   ↓
4. 更新 analysis 表（增加当日计数）
   ↓
5. 继续传递给下一个插件
```

### 排行查询流程

```
1. 接收排行关键词
   ↓
2. 根据时间范围构建 SQL 过滤条件
   ↓
3. 从 analysis 表查询聚合数据
   ↓
4. 查询每个成员的消息类型分布
   ↓
5. 构建排行榜合并转发消息
   ↓
6. 发送给群聊
```

### 详情查询流程

```
1. 接收"发言详情"关键词
   ↓
2. 识别发送者身份
   ↓
3. 分别查询各时间段的统计数据
   - 今日详情
   - 昨日详情
   - 本周详情
   - 本月详情
   - 总结详情
   ↓
4. 格式化并发送（@发送者）
```

## 时间范围说明

| 关键词 | 时间范围 | SQL 过滤条件 |
|-------|---------|------------|
| 今日排行 | 今天 00:00 至当前时刻 | `date = date('now', 'localtime')` |
| 昨日排行 | 昨天 00:00 至 23:59 | `date = date('now', '-1 day', 'localtime')` |
| 本周排行 | 本周一 00:00 至当前时刻 | 从周一开始计算 |
| 本月排行 | 本月 1 号 00:00 至当前时刻 | `date >= date('now', 'localtime', 'start of month')` |
| 总排行 | 所有历史数据 | 无过滤条件 |

**注意**：
- 时间使用服务器本地时区
- 本周从周一开始（ISO 8601 标准）
- 跨天统计在 00:00 时刻分界

## 插件行为

### 优先级和执行顺序

```toml
Priority = -1 << 31  # 最高优先级（最小值，最先执行）
Next = true          # 继续传递给下一个插件
AlwaysRun = true     # 始终运行，即使其他插件已处理
```

**特点**：
- **最先执行**：优先级最高，在所有插件之前运行
- **透明记录**：记录消息后返回 `handled=recorded`，`Next=true` 继续传给后续插件（如 ai），不影响其他插件
- **排行消费**：排行关键词返回 `handled=true`，但因 `Next=true` 仍会传给后续插件（历史行为；如不希望 ai 对排行词回复可改 `Next=false`）
- **全量统计**：`AlwaysRun=true` 保证会话被劫持时仍记录所有群聊消息

### 事件订阅

插件订阅 `message` 事件，自动接收所有消息。

### 消息处理逻辑

```go
if 是排行关键词:
    查询并发送排行榜
    返回 true（已处理，但仍传递给下一个插件）
else:
    记录消息到数据库
    返回 recorded（记录成功则为 true）
```

## 性能考虑

### 数据库优化

- **批量插入**：使用 `INSERT OR IGNORE` 和 `ON CONFLICT` 减少冲突
- **索引加速**：sender 字段建立索引
- **分库存储**：聚合和明细分开，互不影响

### 查询优化

- **聚合优先**：排行查询使用 analysis 表，避免全表扫描
- **限制返回**：默认只返回 Top 10，减少数据传输
- **缓存连接**：数据库连接复用，减少开销

### 存储空间

假设一个 500 人的活跃群，每天产生 1000 条消息：

- **analysis 表**：约 500 行/天（每人每天每类型一行），约 50 KB/天
- **statistics 表**：1000 行/天，约 500 KB/天

一年存储约 **200 MB**，存储压力较小。

## 常见问题

### 1. 排行榜为什么是空的？

**可能原因**：
- 插件刚启用，还没有记录数据
- 查询的时间范围内没有消息
- 数据库文件损坏

**解决方法**：
- 等待一段时间后再查询
- 尝试查询"总排行"
- 检查数据库文件是否存在

### 2. 为什么我发的消息没有统计？

**检查清单**：
- 确认在群聊中发送（不支持私聊统计）
- 确认插件已启用
- 查看日志是否有错误

### 3. 如何清空统计数据？

直接删除数据库文件即可：
```bash
rm analysis.db statistics.db
```

插件会在下次启动时自动重建数据库。

### 4. 排行榜最多显示多少人？

默认显示 Top 10，这是代码中的常量 `defaultRankLimit`。

### 5. 私聊消息会被统计吗？

不会。插件只统计群聊消息（`contact.ContactType_CONTACT_TYPE_CHATROOM`），私聊消息会被忽略。

### 6. 统计数据会占用很多空间吗？

一般不会。一个中等活跃度的群，一年的数据大约 200 MB 左右。如果担心空间问题，可以定期备份并清空旧数据。

### 7. 排行关键词可以自定义吗？

目前不支持。关键词在代码中硬编码：
- 今日排行
- 昨日排行
- 本周排行
- 本月排行
- 总排行
- 发言详情

### 8. 能否导出统计数据？

数据库是标准的 SQLite 格式，可以使用任何 SQLite 工具导出：

```bash
# 导出为 CSV
sqlite3 statistics.db "SELECT * FROM statistics" -csv > messages.csv

# 导出为 JSON
sqlite3 statistics.db "SELECT * FROM analysis" -json > analysis.json
```

## 注意事项

### 1. 隐私保护

- 插件会记录所有群聊消息内容
- 数据库文件包含敏感信息，注意保护
- 在公开环境使用时需告知成员

### 2. 存储管理

- 定期检查数据库文件大小
- 超大群（1000+ 人）需要注意存储空间
- 可以定期备份并清空旧数据

### 3. 性能影响

- 插件会拦截所有消息进行记录
- 对于超大群，可能有轻微性能影响
- 数据库操作是异步的，不会阻塞消息处理

### 4. 数据准确性

- 插件启用前的消息不会被统计
- 插件禁用期间的消息不会被记录
- 确保插件持续运行以保证数据完整性

### 5. 时区问题

- 统计使用服务器本地时区
- 如果服务器时区不正确，统计时间可能有偏差
- 建议确保服务器时区设置正确

## 技术细节

### 并发安全

插件使用 `sync.Mutex` 保护数据库操作，确保并发安全：

```go
p.mu.Lock()
defer p.mu.Unlock()
```

### 错误处理

- 数据库连接失败：插件启动失败
- 记录消息失败：记录警告日志，不影响消息传递
- 查询失败：返回错误信息给用户

### 成员信息获取

插件会尝试获取成员的完整信息（昵称、群昵称）：
1. 优先使用消息中的成员信息
2. 如果可用，调用 chatroom ability 查询
3. 显示优先级：群昵称 [昵称] > 群昵称 > 昵称 > username

### XML 转义

合并转发消息的 XML 内容会自动转义特殊字符，防止 XML 注入。
